# Requirements Document

## Introduction

Enterprise environments often restrict the installation of GitHub Apps on their GitHub organizations. This feature adds an alternative authentication method — Personal Access Token (PAT) authentication — that allows enterprise users to access private repositories without requiring a GitHub App to be installed on their organization. The PAT-based flow lets users provide their own GitHub token, which the backend stores in a session and uses to proxy GitHub API requests, reusing the existing session and proxy infrastructure.

## Glossary

- **Backend**: The PHP server application that handles authentication, session management, and proxies GitHub API requests
- **Frontend**: The React/TypeScript single-page application that renders markdown content
- **PAT**: A GitHub Personal Access Token generated by the user from their GitHub account settings
- **Session_Manager**: The server-side component responsible for creating, validating, and destroying sessions
- **OAuth_Flow**: The existing GitHub App OAuth authentication mechanism
- **Auth_Router**: The backend routing layer that dispatches authentication-related HTTP requests
- **Auth_Service**: The frontend service responsible for managing authentication state and communicating with the backend
- **Token_Validator**: The backend component that verifies a PAT is valid by calling the GitHub API

## Requirements

### Requirement 1: PAT Login Endpoint

**User Story:** As an enterprise user, I want to authenticate using a Personal Access Token, so that I can access private repositories without requiring a GitHub App installation on my organization.

#### Acceptance Criteria

1. WHEN a POST request is received at `/api/auth/pat-login` with a JSON body containing a `token` field, THE Backend SHALL validate the token by calling the GitHub API `/user` endpoint with the provided token, using a request timeout of 10 seconds
2. WHEN the GitHub API confirms the token is valid, THE Backend SHALL create a session storing the PAT as the installation_token and return a JSON response with `{ "authenticated": true }`
3. WHEN the GitHub API confirms the token is valid, THE Backend SHALL set an httpOnly, Secure, SameSite=Strict session cookie with a 1-hour expiry
4. IF the request body is not valid JSON, or the `token` field is missing, empty, or not a string, THEN THE Backend SHALL respond with HTTP 400 and a JSON error message indicating that a valid token is required
5. IF the GitHub API returns an unauthorized response for the provided token, THEN THE Backend SHALL respond with HTTP 401 and a JSON error message indicating that the token is invalid
6. IF the GitHub API does not respond within 10 seconds or is unreachable, THEN THE Backend SHALL respond with HTTP 502 and a JSON error message indicating that the token could not be verified
7. IF the provided token exceeds 256 characters in length, THEN THE Backend SHALL respond with HTTP 400 and a JSON error message indicating that the token format is invalid

### Requirement 2: Session Compatibility

**User Story:** As a developer, I want PAT-based sessions to reuse the existing session infrastructure, so that proxy routes and session management work identically regardless of authentication method.

#### Acceptance Criteria

1. THE Session_Manager SHALL store PAT-based sessions in the same file-based JSON format as OAuth-based sessions, including the fields `installation_token`, `auth_method`, `created_at`, and `expires_at`.
2. THE Session_Manager SHALL include an `auth_method` field in session data with value `"pat"` for PAT-based sessions and `"oauth"` for OAuth-based sessions.
3. WHEN a PAT-based session is created, THE Session_Manager SHALL set the `installation_token` field to the user-provided PAT value and SHALL set `expires_at` to 24 hours from the creation time.
4. THE proxy routes SHALL authorize and forward requests from PAT-based sessions using the same `installation_token` extraction, scope checking, and error response logic as OAuth-based sessions, without requiring any conditional branching on `auth_method`.
5. IF a PAT-based session creation is attempted with an empty or whitespace-only PAT value, THEN THE Session_Manager SHALL reject the request and return an error indicating that a valid PAT is required.

### Requirement 3: Frontend PAT Login Interface

**User Story:** As an enterprise user, I want a PAT login option in the application interface, so that I can choose to authenticate with a Personal Access Token instead of the OAuth flow.

#### Acceptance Criteria

1. WHEN the user is not authenticated and private access is available, THE Frontend SHALL display both an OAuth login option and a PAT login option.
2. WHEN the user selects PAT login, THE Frontend SHALL display a password-masked text input field (with `type="password"`) for entering a Personal Access Token, accepting a maximum of 255 characters.
3. IF the user attempts to submit the PAT input while the field is empty or contains only whitespace, THEN THE Frontend SHALL disable the submit action and not send a request to the backend.
4. WHEN the user submits a non-empty PAT, THE Auth_Service SHALL send a POST request to `/api/auth/pat-login` with the token in the request body.
5. WHEN the PAT login endpoint returns `{ "authenticated": true }`, THE Auth_Service SHALL update the local authentication state to authenticated.
6. IF the PAT login endpoint returns a response with an HTTP status code of 4xx or 5xx, THEN THE Frontend SHALL display the error message from the response body to the user.
7. IF a network error or timeout occurs during the PAT login request, THEN THE Frontend SHALL display an error message indicating that the server could not be reached.
8. THE Frontend SHALL provide a link to the GitHub documentation for creating Personal Access Tokens, visible alongside the PAT input field.

### Requirement 4: Auth Status Enhancement

**User Story:** As a user, I want the auth status endpoint to indicate the authentication method used, so that the frontend can display relevant session information.

#### Acceptance Criteria

1. WHEN a valid session exists that was created via the OAuth flow, THE Backend SHALL return a JSON response from `/api/auth/status` containing `{ "authenticated": true, "auth_method": "oauth" }`.
2. WHEN a valid session exists that was created via a Personal Access Token, THE Backend SHALL return a JSON response from `/api/auth/status` containing `{ "authenticated": true, "auth_method": "pat" }`.
3. WHEN a valid scoped session exists (created via the share flow), THE Backend SHALL return a JSON response from `/api/auth/status` containing `{ "authenticated": true, "auth_method": "oauth" }` reflecting the original authentication method used to create the shared session.
4. WHEN no valid session exists, THE Backend SHALL return `{ "authenticated": false }` without an `auth_method` field.
5. IF the session cookie is present but references an expired or invalid session, THEN THE Backend SHALL return `{ "authenticated": false }` without an `auth_method` field.

### Requirement 5: PAT Security Constraints

**User Story:** As a security-conscious user, I want PAT authentication to follow secure practices, so that my token is protected during and after the login process.

#### Acceptance Criteria

1. IF a PAT login request is received and the Backend is running in production mode (determined by the environment configuration variable being set to "production"), THEN THE Backend SHALL reject the request with an error response indicating that a secure connection is required, unless the request was made over HTTPS.
2. THE Backend SHALL never include the PAT value in error response bodies, server log output, or any other system output observable outside the running process.
3. WHEN a PAT-based session expires (after 1 hour from creation), THE Session_Manager SHALL delete the session file containing the stored token from disk.
4. WHEN the user submits the PAT login form (regardless of whether authentication succeeds or fails), THE Frontend SHALL clear the PAT input field value before the next browser render frame completes.
5. THE Frontend SHALL transmit the PAT only in the request body via the existing credentialed fetch mechanism and SHALL NOT include the PAT in URL parameters, request headers other than the body, or browser storage (localStorage, sessionStorage, or cookies).
6. THE Frontend SHALL NOT persist the PAT value in localStorage, sessionStorage, cookies, or any other client-side storage mechanism at any point during or after the login process.

### Requirement 6: PAT Repository Restriction

**User Story:** As an enterprise user, I want to optionally restrict my PAT session to specific repositories, so that the viewer application only accesses the repositories I intend to use.

#### Acceptance Criteria

1. WHEN a POST request to `/api/auth/pat-login` includes an optional `scope` field with `owner` and `repo` values, THE Backend SHALL create a scoped session restricted to that repository, where all proxy endpoint requests (contents and raw) are limited to paths within the specified owner/repo.
2. IF a scoped PAT session attempts to access a repository outside the specified scope, THEN THE Backend SHALL respond with HTTP 403 and a JSON error message indicating that the session is restricted to a specific repository path.
3. WHEN no `scope` field is provided in the PAT login request, THE Backend SHALL create an unscoped session that can access any repository the token has permission for.
4. IF the `scope` field is provided but does not contain both a non-empty `owner` and a non-empty `repo` value, THEN THE Backend SHALL respond with HTTP 400 and a JSON error message indicating the scope format is invalid.
5. WHEN the user selects PAT login, THE Frontend SHALL display an optional text input field to specify a repository in `owner/repo` format for scope restriction, with a maximum input length of 256 characters.
6. IF the user enters a value in the repository restriction field that does not match the `owner/repo` format (exactly one forward slash separating two non-empty segments), THEN THE Frontend SHALL display a validation error and prevent form submission.
7. THE Frontend SHALL display helper text adjacent to the repository restriction field indicating that the field is optional and that GitHub fine-grained tokens provide additional server-side scoping.

### Requirement 7: Logout Compatibility

**User Story:** As a user who logged in via PAT, I want the logout flow to work the same as OAuth logout, so that my session is properly invalidated.

#### Acceptance Criteria

1. WHEN a user with a PAT-based session sends a POST request to `/api/auth/logout`, THE Backend SHALL destroy the session file and clear the session cookie, returning a JSON response with `{ "authenticated": false }`.
2. THE existing logout endpoint SHALL handle both PAT-based and OAuth-based sessions without requiring changes to its interface or request format.
3. IF a logout request is received with no valid session or an already-expired session, THEN THE Backend SHALL still return a success response with `{ "authenticated": false }` and clear any session cookie present.
